## 一、通用 Agent 的核心能力

### 1. 代码理解与编辑

- 阅读、搜索整个代码库（语义搜索、关键词搜索、按路径查找）
- 修改、创建、删除文件
- 遵循项目现有风格与约定，做**最小必要改动**
- 解释代码逻辑、架构、数据流

### 2. 终端与命令执行

- 运行 shell 命令（构建、测试、安装依赖、Git 操作等）
- 在沙箱中执行，必要时申请网络或更高权限
- 监控后台任务与终端输出

### 3. 调试与排错

- 分析报错、日志、CI 失败
- 定位根因并修复
- 查看 linter / 诊断信息

### 4. Git 与协作

- `status` / `diff` / `log` 等查看变更
- 按你的要求创建 commit（不会擅自提交）
- 用 `gh` 创建 PR、查 issue、看 CI

### 5. 研究与信息获取

- 联网搜索技术文档、版本信息
- 抓取网页内容
- 在代码库内做定向探索

### 6. 任务规划与执行

- 复杂任务拆步、跟踪进度
- 多文件、多步骤协同修改
- 在实现 / 规划 / 调试模式间切换

### 7. 专业技能（Skills）

- 按任务自动选用技能（Vue、Docker、OpenSpec、前端设计、安全审查等）
- 技能提供领域最佳实践与固定工作流

### 8. 子 Agent 委派

复杂任务可启动专门子 Agent，例如：

| 子 Agent            | 用途                     |
| ------------------- | ------------------------ |
| **generalPurpose**  | 通用研究、搜索、多步任务 |
| **explore**         | 快速只读探索代码库       |
| **shell**           | 命令执行、Git 操作       |
| **cursor-guide**    | Cursor 产品使用说明      |
| **ci-investigator** | 分析 PR CI 失败原因      |
| **bugbot**          | 代码变更审查             |
| **security-review** | 安全审查                 |

### 9. 交互与产出

- 用代码引用格式指向具体文件与行号
- 用 Mermaid / ASCII 画架构与流程
- 需要时用 Canvas 做可视化分析
- 结构化提问（多选）以澄清需求

### 10. 图像

- 读取你提供的图片
- 在你明确要求时生成图片

## 二、能力边界（不会擅自做）

- **不主动 commit / push**，除非你明确要求  
- **不擅自扩大范围**，只做你要求或任务必需的事  
- **不跳过 Git hooks**，除非你明确要求  
- **不提交含密钥的文件**  
- 沙箱下部分操作需额外权限（如广域网络）

## 三、`generalPurpose` 子 Agent 的定位

`generalPurpose` 是「通用子 Agent」，适合：

- 需要多步自主完成的复杂任务  
- 跨模块搜索与研究  
- 父 Agent 一次难以覆盖的大范围探索  

与主 Agent 相比，它更偏**在子任务里自主跑完一段流程**，再把结果汇总回来。

## 1. 先明确「循环调优」调什么

常见有三类，建议分开设计，不要混在一个 loop 里：

| 类型           | 调优对象                         | 典型场景                 |
| -------------- | -------------------------------- | ------------------------ |
| **任务循环**   | 单次任务输出质量                 | 写代码、写报告、多步推理 |
| **工作流循环** | Agent 编排（步骤、工具、分支）   | 复杂业务流程             |
| **平台循环**   | Prompt、工具描述、模型参数、策略 | 长期提升成功率           |

核心闭环可以统一抽象为：

```
Observe → Evaluate → Diagnose → Propose → Apply → Verify → (继续/结束)
```

## 3. 核心抽象：把一次循环变成可编排对象

建议平台统一这几类对象：

### 3.1 `Run`（一次执行）

```json
{
  "run_id": "run_001",
  "agent_version": "v3",
  "input": { "task": "生成登录接口", "context": {} },
  "trace": [],
  "output": {},
  "status": "success | failed | partial",
  "cost": { "tokens": 12000, "latency_ms": 4500 }
}
```

### 3.2 `Evaluation`（评估结果）

```json
{
  "run_id": "run_001",
  "score": 0.72,
  "passed": false,
  "dimensions": {
    "correctness": 0.6,
    "completeness": 0.8,
    "safety": 1.0
  },
  "failures": [
    { "type": "missing_validation", "evidence": "未校验密码长度" }
  ],
  "feedback": "需要补充输入校验与错误码定义"
}
```

### 3.3 `Patch`（优化提案）

```json
{
  "target": "system_prompt | tool_schema | workflow_step | model_config",
  "change": { "...": "..." },
  "rationale": "上一轮缺少校验逻辑",
  "risk": "low"
}
```

### 3.4 `LoopSession`（一次调优会话）

```json
{
  "session_id": "loop_abc",
  "goal": "让登录接口生成成功率 >= 90%",
  "baseline_version": "v3",
  "current_version": "v3.2",
  "iterations": [
    { "run_id": "...", "eval_id": "...", "patch_id": "..." }
  ],
  "stop_reason": "budget_exceeded | target_met | no_improvement"
}
```

有了这四个对象，循环调优就可以平台化，而不是写死在某个业务里。

## 4. 循环编排器（Loop Orchestrator）设计

### 4.1 标准循环伪代码

```python
def agent_tune_loop(session):
    version = session.baseline_version
    
    for i in range(session.max_iters):
        run = execute_agent(version, session.input, budget=session.budget)
        eval_result = evaluate(run, session.criteria)
        
        record(session, run, eval_result)
        
        if should_stop(eval_result, session):
            return finalize(session, version, "target_met")
        
        diagnosis = diagnose(run, eval_result)          # 失败归因
        patches = propose_changes(diagnosis, version)   # 生成候选修改
        
        best_patch = select_patch(patches, session.policy)
        candidate = apply_patch(version, best_patch)
        
        # 关键：先沙箱验证，再晋升
        verify = sandbox_verify(candidate, session.test_cases)
        if verify.improved:
            version = promote(candidate)
        else:
            version = rollback_or_keep(version)
        
        if no_improvement(session.history, window=3):
            return finalize(session, version, "no_improvement")
    
    return finalize(session, version, "budget_exceeded")
```

### 4.2 停止策略（必须有）

避免无限循环，建议同时满足多条：

- **目标达成**：`score >= threshold` 且关键维度全过
- **预算耗尽**：token / 时间 / 迭代次数
- **收益停滞**：连续 N 轮提升 < ε
- **风险阈值**：改动过大或评测波动异常
- **人工中断**：HITL 审批点

---

## 5. 评估层怎么设计（决定调优是否有效）

### 5.1 评估器分层

| 层级         | 方式                        | 优点         | 缺点     |
| ------------ | --------------------------- | ------------ | -------- |
| L1 规则      | 单元测试、schema 校验、lint | 稳定、可复现 | 覆盖有限 |
| L2 程序化    | 执行结果对比、API 回放      | 客观         | 需测试集 |
| L3 LLM Judge | rubric 打分 + 证据引用      | 灵活         | 有波动   |
| L4 人工      | 抽检 / 关键节点审批         | 最准         | 成本高   |

推荐：**L1/L2 做 gate，L3 做细粒度反馈，L4 做高风险场景兜底**。

### 5.2 Rubric 示例（给 Judge）

```yaml
criteria:
  - name: correctness
    weight: 0.5
    checks:
      - "是否满足功能需求"
      - "是否有明显逻辑错误"
  - name: robustness
    weight: 0.3
    checks:
      - "边界条件处理"
      - "错误处理完整性"
  - name: maintainability
    weight: 0.2
    checks:
      - "结构清晰"
      - "命名合理"
```

评估输出必须带 **evidence**（引用 trace 中的步骤/工具调用/输出片段），否则无法做有效归因。

## 6. 优化层：改什么、怎么改

### 6.1 可调参对象（建议版本化）

- `system_prompt` / `developer_prompt`
- `tool descriptions` / `tool selection policy`
- `workflow graph`（步骤顺序、分支条件）
- `retrieval config`（topK、rerank）
- `model routing`（模型、温度、max tokens）
- `memory policy`（短期记忆窗口、摘要策略）

每一项都进 **Version Registry**，支持 diff、回滚、A/B。

### 6.2 归因 → 提案映射（很实用）

| 失败类型   | 典型补丁                        |
| ---------- | ------------------------------- |
| 工具选错   | 优化 tool schema / 增加示例     |
| 步骤遗漏   | workflow 增加 mandatory step    |
| 格式错误   | 加强 output schema + 自动修复器 |
| 幻觉       | 加强 grounding / 强制检索       |
| 过度冗长   | 压缩 prompt / 降 max_tokens     |
| 不安全输出 | 加 guardrail + refusal 模板     |

### 6.3 一次只改一个变量（强烈建议）

否则无法判断哪项改动有效。平台层可做：

```text
Patch Policy: single-knob-per-iteration
```

## 7. 沙箱与晋升机制（避免「越调越差」）

每次 patch 不要直接上线，走：

```
Candidate → Shadow Run → Compare with Baseline → Promote/Rollback
```

- **Shadow Run**：同一批 test cases 跑 baseline 与 candidate
- **晋升条件**：`candidate_score - baseline_score >= delta` 且关键 gate 全过
- **回滚**：任一安全项失败立即回滚

测试集建议分两套：

- **开发集**（调优时可看，防过拟合也要控制）
- **_holdout 集**（只在最终晋升时用）

---

## 8. 可观测性：没有 trace 就没有调优

每次循环至少记录：

- 输入、输出、工具调用链
- token / latency / 成本
- 每轮 eval 分数与失败标签
- patch 内容与前后 diff
- 晋升/回滚决策原因

建议提供「Loop Timeline」视图：

```text
Iter1: score 0.55 -> patch(prompt+example) -> Iter2: 0.68
Iter2: score 0.68 -> patch(tool_schema)     -> Iter3: 0.67 (rollback)
Iter3: score 0.68 -> patch(workflow_step)   -> Iter4: 0.81 (promote)
```

---

## 9. 人机协同（HITL）建议放哪

全自动循环适合低风险任务；高风险场景在三个点加人：

1. **Patch 审批**：改 system prompt / 工具权限前
2. **晋升审批**：新版本影响生产流量前
3. **失败复盘**：连续停滞时人工标注根因，反哺规则库

```text
POST /diagnostics            失败归因
POST /patches/propose          基于诊断生成补丁
```

Agent 执行层保持你们现有接口，调优层通过 **version pin** 调用：

```text

```

## 11. 推荐落地路径（分三期）

### Phase 1：任务循环（2–3 周）

- Run / Trace / Evaluation
- 固定 rubric + 规则 gate
- 手动 patch（人工改 prompt）+ 自动重跑

### Phase 2：半自动调优（3–5 周）

- 自动 diagnose + propose
- 沙箱对比 + 版本晋升/回滚
- Loop Orchestrator + stop policy

### Phase 3：平台化调优（持续）

- 多 agent、多目标优化
- 测试集管理、实验平台（A/B）
- 成本/成功率联动优化

---

## 12. 常见坑

1. **没有客观 gate，只靠 LLM 打分** → 容易自嗨循环  
2. **每轮改太多变量** → 无法归因  
3. **没有 holdout** → 过拟合测试样例  
4. **没有预算上限** → 成本失控  
5. **trace 不完整** → 无法解释「为什么变好/变差」  
6. **直接把调优结果上线** → 必须沙箱 + 晋升流程  

---

## 13. 一个最小可行方案（MVP）

如果你们想最快验证价值，MVP 可以只做：

1. 固定 20 条 benchmark 任务  
2. 每轮：执行 → 规则评测（通过率）→ LLM 给出 3 条改进建议  
3. 人工选 1 条建议改 prompt  
4. 自动重跑并对比分数  
5. 分数提升才保存新版本  

这已经是一个可用的「agent 循环调优」闭环，之后再逐步自动化 diagnose/propose/promote。

## 1. 先分清两层循环

```mermaid
flowchart TB
    subgraph Outer["外层：调优循环 Tune Loop"]
        T1[选版本/配置] --> T2[跑 benchmark]
        T2 --> T3[评估]
        T3 --> T4[归因 + 提案]
        T4 --> T5[改 MCP/API/工作流/Agent/Skill]
        T5 --> T1
    end

    subgraph Inner["内层：执行循环 Execution Loop（你们已有）"]
        I1[模型规划 Step 1] --> I2[Step 2]
        I2 --> I3[Step 3]
        I3 --> I4[Step 4]
        I4 --> I5[Step 5 上限]
        I5 --> I6[模型总结输出]
    end

    T2 --> Inner
    I6 --> T3
```

| 层级     | 作用            | 次数                         | 调什么                                 |
| -------- | --------------- | ---------------------------- | -------------------------------------- |
| **内层** | 完成单次任务    | ≤ 5 步规划                   | 不改配置，只执行                       |
| **外层** | 提升成功率/质量 | 你们平台控制（建议 3–10 轮） | 改 MCP/API/工作流/Agent/Skill/规划策略 |

原则：**内层负责「这次怎么做」，外层负责「下次用什么配置做得更好」。**

## 2. 把你们平台能力映射成「可调对象」

总 Agent 是一个**编排容器**，里面挂的资源都可以版本化、可 diff、可回滚：

```text
MasterAgent (v1.3)
├── PlannerConfig      # 规划策略（步数上限、是否允许跳步、总结模板）
├── Agents[]           # 子 Agent
├── Skills[]           # Skill 绑定与优先级
├── Workflows[]        # 工作流节点/分支
├── MCPs[]             # MCP 工具集
└── APIs[]             # 外部 API 定义
```

### 每层适合调什么

| 组件         | 常见失败           | 调优动作                          |
| ------------ | ------------------ | --------------------------------- |
| **Skill**    | 领域规范没生效     | 换 Skill、调优先级、补约束        |
| **Agent**    | 子任务分工错       | 拆/合 Agent、改职责描述           |
| **Workflow** | 步骤顺序/分支错    | 改 DAG、加 mandatory 节点         |
| **MCP**      | 工具选错/参数错    | 改 tool 描述、过滤工具、加示例    |
| **API**      | 调用失败/字段错    | 改 schema、重试策略、mock 兜底    |
| **Planner**  | 5 步内做不完或空转 | 改规划 prompt、步数分配、早停规则 |

建议平台约束：**每轮外层调优只改 1 类组件**（例如只改 Skill，或只改 Workflow），否则无法归因。

## 3. 内层 5 步规划：怎么给外层提供信号

你们最后有「模型总结输出」，这是调优的关键输入，但要补**结构化轨迹**，否则只能人工看总结。

### 3.1 每次执行建议落库

```json
{
  "run_id": "run_xxx",
  "master_agent_version": "v1.3",
  "plan_trace": [
    {
      "step": 1,
      "intent": "分析需求并选择 Skill",
      "selected": { "type": "skill", "id": "vue-best-practices" },
      "tool_calls": [],
      "status": "ok"
    },
    {
      "step": 3,
      "intent": "调用 MCP 查库表",
      "selected": { "type": "mcp", "id": "db-mcp" },
      "tool_calls": [{ "name": "query", "args": {}, "error": "timeout" }],
      "status": "failed"
    }
  ],
  "steps_used": 4,
  "summary": "最终总结文本...",
  "artifacts": { "output": "...", "files": [] }
}
```

### 3.2 评估不要只看 summary

1. **硬规则**：API 成功率、workflow 是否跑完、是否超 5 步、超时

3. **总结质检**：summary 是否覆盖结论、证据、未完成项


## 4. 外层调优循环（贴合你们架构）

```python
def tune_master_agent(session):
    config = session.baseline_config  # MasterAgent 全量配置快照

    for iter in range(session.max_tune_iters):
        runs = [execute_master(config, case) for case in session.benchmarks]
        evals = [evaluate(run) for run in runs]
        aggregate = aggregate_scores(evals)

        if aggregate.passed:
            return promote(config, reason="target_met")

        diagnosis = diagnose(runs, evals)  
        # 例: "step3 频繁选错 MCP", "5步内未触发 workflow"

        patch = propose_single_knob_patch(diagnosis, config)
        candidate = apply_patch(config, patch)

        shadow = compare(
            baseline=session.baseline_config,
            candidate=candidate,
            benchmarks=session.holdout_cases
        )

        if shadow.improved:
            config = candidate
            session.baseline_config = candidate  # 滚动提升
        else:
            log_rollback(patch)

        if stagnation(session.history, n=3):
            break

    return finalize(config)
```

## 5. Tune Engine 循环（你要实现的核心）

```python
def auto_tune(target_id, input_payload=None):
    target = load_target(target_id)
    best = {
        "version": target.entry_ref,
        "score": -1,
        "artifact": None
    }

    for i in range(target.constraints.max_tune_iters):
        # 1) 跑 benchmark
        runs = run_benchmark(target.entry_ref, target.benchmark_suite_id)

        # 2) 评估
        eval_result = evaluate(runs, target.evaluator_id)
        score = eval_result.aggregate_score

        # 3) 记录并更新最优
        if score > best["score"]:
            best = {
                "version": target.entry_ref,
                "score": score,
                "artifact": eval_result.best_run_output
            }

        # 4) 达标 -> 直接返回最优
        if meets_success_criteria(eval_result, target.success_criteria):
            return {
                "status": "optimal",
                "best_version": best["version"],
                "score": best["score"],
                "result": best["artifact"],
                "iters": i + 1
            }

        # 5) 未达标 -> 归因 + 生成补丁（每轮只改一个旋钮）
        diagnosis = diagnose(runs, eval_result)
        patch = propose_patch(diagnosis, target.patchable_components)

        # 6) 沙箱验证补丁
        candidate_ref = apply_patch(target.entry_ref, patch)
        shadow = shadow_compare(
            baseline=target.entry_ref,
            candidate=candidate_ref,
            suite=target.benchmark_suite_id
        )

        if shadow.improved:
            target.entry_ref = promote(candidate_ref)
        else:
            discard(candidate_ref)

        # 7) 停滞/预算 -> 返回当前最优（未必达标）
        if stagnation(i, score) or budget_exceeded():
            break

    return {
        "status": "best_effort",
        "best_version": best["version"],
        "score": best["score"],
        "result": best["artifact"],
        "message": "未达目标阈值，返回当前最优"
    }
```

你要的「最优返结果，不最优继续循环」就是这里的 **第 4 步 + 第 7 步**。

## 6. MCP 怎么放：给总 Agent 用的工具集

建议 MCP 名：`tune-mcp`，工具如下：

| 工具                   | 作用                                      |
| ---------------------- | ----------------------------------------- |
| `tune.start`           | 启动调优会话（传 `target_id` 或自动识别） |
| `tune.status`          | 查当前迭代、分数、最优版本                |
| `tune.get_result`      | 取最优结果（`optimal` / `best_effort`）   |
| `tune.stop`            | 人工中止                                  |
| `tune.register_target` | 注册新技能到调优体系（可选，管理面）      |

总 Agent 不需要自己写循环，只负责：

1. 识别这是调优任务  
2. 调 `tune.start`  
3. 轮询或等待 `tune.get_result`  
4. 把最终结果总结给用户

## 7. Skill 怎么放：`auto-tune` 通用技能

Skill 不写循环代码，只定义**模式识别 + 行为规范**：

```markdown
# auto-tune skill

## 何时进入调优模式
- 用户说：调优 / 优化 / 提升到xx准确率 / 自动迭代
- 或任务带参数：mode=auto_tune

## 执行规范
1. 识别 target（如 contract-parse）
2. 调用 MCP tune.start
3. 未返回 optimal 前，不手工改业务配置
4. 最终只输出：
   - 是否达标（optimal / best_effort）
   - 最优版本号
   - 关键指标
   - 最终结果
   - 若未达标，给下一步建议
```

以后任何新技能，只要在总 Agent 里挂上这个 Skill，就能走同一套调优模式。

## 8. 工作流怎么放：通用调优模板

做一个平台级工作流 `WF_TUNE_GENERIC`，所有调优都走它：

```text
1. 解析目标 target_id
2. tune.start
3. 等待引擎完成（或分段 status 轮询）
4. tune.get_result
5. 模型总结输出（达标情况、版本、指标、结果）
```

业务工作流（如合同解析）**不改动**，只作为 `entry_ref` 被 Tune Engine 反复调用。

```text
WF_TUNE_GENERIC
  └─ 反复调用 -> wf_contract_parse@vN
```

## 9. 怎么做到「有其他技能时自动走调优模式」

靠 **路由规则**，不靠人肉判断：

### 规则 1：意图识别（总 Agent 规划 Step1）

- 关键词：`调优`、`优化到`、`自动迭代`、`benchmark`
- 或结构化参数：`{"mode":"auto_tune","target":"xxx"}`

### 规则 2：Target 映射表

```text
合同解析   -> target_id = contract-parse
发票识别   -> target_id = invoice-parse
代码审查   -> target_id = code-review
```

### 规则 3：强制走调优工作流

总 Agent 规划时：

```text
if mode == auto_tune:
    route_to(WF_TUNE_GENERIC, target_id=xxx)
else:
    route_to(业务工作流)
```

### 规则 4：Evaluator 插件化

每个 target 绑定自己的 evaluator，但接口统一：

```text
evaluate(run_output, ground_truth) -> {score, passed, failures[]}
```

## 10. 「最优」怎么定义（必须配置化）

每个 target 都配 `success_criteria`，例如：

```yaml

  min_pass_rate: 0.90
  min_field_f1: 0.88
  max_error_rate: 0.05

  min_pass_rate: 0.95
```

引擎判断：

- 全满足 → `status=optimal`，立即返回  
- 不满足 → 继续循环  
- 到上限仍不满足 → `status=best_effort`，返回当前最优

**Cursor**

**Tunable Target 协议**，说白了就是一个**约定好的「说明书格式」**：  
告诉调优引擎——**要调谁、怎么测、什么叫好、能改什么、什么时候停**。

---

## 用一句话理解

把你的某个能力（比如「合同解析工作流」）登记成调优对象时，按统一格式填一份配置，这份配置就是 **Tunable Target**；  
**协议**就是大家都按这套字段来，引擎才能通用处理，不用每个技能各写一套调优逻辑。

---

## 类比

像快递下单的标准面单：

| 面单字段   | 调优协议字段                     |
| ---------- | -------------------------------- |
| 收件人地址 | 调哪个 Agent / 工作流            |
| 物品类型   | 类型（workflow / agent / skill） |
| 验收标准   | 什么叫「最优」                   |
| 禁运规则   | 最多循环几次、能改哪些配置       |

有这张「面单」，通用调优引擎就知道怎么处理它。

---

## 协议里通常有什么

```json
{
  "target_id": "contract-parse",
  "type": "workflow",
  "entry_ref": "wf_contract_parse@v12",
  "benchmark_suite_id": "contract_bench_v1",
  "evaluator_id": "contract_field_eval",
  "patchable_components": ["workflow", "skill", "agent"],
  "success_criteria": {
    "min_pass_rate": 0.9
  },
  "constraints": {
    "max_tune_iters": 8
  }
}
```

逐项说明：

| 字段                   | 意思                            |
| ---------------------- | ------------------------------- |
| `target_id`            | 这个可调优对象的唯一名字        |
| `type`                 | 类型：工作流 / Agent / Skill 等 |
| `entry_ref`            | 当前要跑的版本（入口）          |
| `benchmark_suite_id`   | 用哪套测试样本                  |
| `evaluator_id`         | 用哪个评分器判好坏              |
| `patchable_components` | 允许自动改哪些东西              |
| `success_criteria`     | 什么叫「最优」，达标就返回      |
| `constraints`          | 循环上限、成本上限等            |

---

## 它解决什么问题

没有协议时：

```text
合同解析一套调优逻辑
发票识别又一套
代码审查再一套
→ 不可复用
```

有协议后：

```text
Tune Engine 永远只做一件事：
  拿 target 配置 -> 跑测 -> 评分 -> 改配置 -> 循环

新技能只要新增一份 target 配置
```

这就是「通用循环调优」能成立的关键。

```json
  "benchmark_suite_id": "contract_30_cases",
```

```text
    │    "plan_trace": [...]   // 有更好，没有也行
    │  }
    ▼
Tune Engine
    解析 answer → 和标准答案比对 → 打分 → 决定要不要继续调优
```

**你不需要给 Tune Engine 单独的工作流执行接口**，只要能把 query 交给 Agent，并拿回总结后的 `answer` 即可。

## 六、自动调优时，实际改的是什么

规划模型是按**工作流描述**来选的，所以调优主要动这些配置（都在平台里，由 Tune Engine 通过接口或版本系统去改）：

| 改什么                     | 为什么                             |
| -------------------------- | ---------------------------------- |
| **工作流描述**             | 规划模型按它规划；描述不清会走错路 |
| **工作流内部节点 / Skill** | 真正干活的逻辑                     |
| **工作流里的 MCP / API**   | 工具调用                           |
| **总结模型 prompt**        | 最终答案格式、是否漏字段           |

每一轮**只改一处**，便于判断有没有变好。

## 八、和「仅仅规划到工作流」的关系

你的 Agent **不用负责调优**，只做三件事：

1. **规划**：根据描述，把 query 路由到对应工作流  
2. **等待**：工作流跑完  
3. **总结**：把结果整理成最终答案  

Tune Engine 在外面：

- 用**同一批 query**反复考这个 Agent  
- 根据**总结答案**好不好，决定改工作流还是改描述  
- 达标就停，返回最优配置和结果  

```text
Agent 的职责（不变）：
  规划 → 工作流 → 总结

Tune Engine 的职责（新增）：
  出题 → 调用 Agent → 阅卷 → 改配置 → 再考
```

## 一、现在这套逻辑通用吗？

**部分通用**，要分清适用场景：

| 能力                            | 现在这套（Tune Engine） | 是否通用            |

| 批量测试 + 打分                 | 有                      | 通用                |

| 多技能注册（合同、发票…）       | 有                      | 通用                |

- **离线调优**（用很多测试题，把 Agent/工作流配置越调越好）→ 现在这套**通用**，任何技能只要：测试题 + 评分规则 + 可改配置，就能接进来。

可以记成：

```text
Tune Engine     = 教练（长期训练，改配置）
Agent 内循环    = 运动员当场纠错（同一次请求里重试）
```

两样可以**同时存在**，不冲突。

## 四、单次 query 循环：在平台里怎么实现

在 Agent 里加 3 个能力即可，不必塞进 Tune Engine。

### 1. 执行后检查（Checker）

工作流 + 总结完成后，立刻做快速检查，例如：

```text
- 必填字段齐了吗？
- 输出是合法 JSON 吗？
- 和业务规则一致吗？
```

检查不通过 → 记下 `failure_points`，例如：

```json
{
  "missing_fields": ["amount"],
  "format_error": "sign_date 不是 YYYY-MM-DD",
  "score": 0.6
}
```

### 2. 带问题点重新规划（Replanner）

第二次规划时，不只传原始 query，还要传上次失败信息：

```text
【原始任务】
解析合同，返回 JSON：party_a, party_b, amount, sign_date

【上次执行问题】
- amount 字段缺失
- sign_date 格式错误

【要求】
针对以上问题重新规划并执行，补全缺失字段。
```

规划模型按工作流描述，选路径或换步骤再跑。

### 3. 停止条件

```text
- 检查通过 → 立刻输出最终结果（optimal）
- 已达 5 次规划上限 → 输出目前最好的一次
- 连续两次没改善 → 提前停，避免空转
```

### 流程图

```mermaid
flowchart TD
    A[用户 query] --> B[第N次规划]
    B --> C[工作流执行]
    C --> D[总结模型]
    D --> E{检查通过?}
    E -->|是| F[输出最终结果]
    E -->|否| G[记录 failure_points]
    G --> H{N < 5 且还能改善?}
    H -->|是| B
    H -->|否| I[输出当前最优结果]
```

## 七、你要的「到最优结果输出」怎么定义

两种「最优」要分开：

| 场景                 | 最优含义                              |
| -------------------- | ------------------------------------- |
| **单次 query 循环**  | Checker 通过，或 5 次里分数最高那次   |
| **Tune Engine 调优** | 整批 benchmark 达标（如通过率 ≥ 90%） |

单次 query 的伪代码：

```python
def handle_query(query):
    failure_points = []
    best = None

    for attempt in range(1, 6):  # 最多5次
        plan = planner.plan(query, failure_points, workflow_desc)
        wf_result = workflow.run(plan)
        answer = summarizer.summarize(wf_result)
        check = checker.evaluate(answer, query)

        if check.score > (best.score if best else -1):
            best = {"answer": answer, "score": check.score, "attempt": attempt}

        if check.passed:
            return {"status": "optimal", "answer": answer, "attempt": attempt}

        failure_points = check.failure_points  # 记录不好点，给下次规划

    return {"status": "best_effort", "answer": best.answer, "attempt": best.attempt}
```

这就是你说的：**第一次不好 → 记录不好点 → 规划第二次 → … → 输出最优结果**。

## 三个结合点（怎么串起来）

### 结合点 1：Tune Engine 调好的版本，单次循环直接用

```text
Tune Engine 跑出最优版本 wf_contract_parse@v5
    ↓
在平台把生产 Agent 指向 v5
    ↓
用户发 query 走单次循环时，第一次成功率更高，重试更少
```

**价值**：离线把底子打好，在线少折腾。

---

### 结合点 2：单次循环的失败 case，导入 Tune Engine

单次循环里 `check.evaluate` 返回 `passed=false` 时，会有：

```json
{
  "query": "原始问题",
  "failure_points": { "missing_fields": ["amount"] },
  "output": { ... }
}
```

这些可以整理成 benchmark，给 Tune Engine 长期调优：

```text
线上用户 query 失败
    ↓
记录 query + failure_points + 标准答案（人工补或半自动）
    ↓
导入 Tune Engine benchmark
    ↓
定期 tune.start → 自动改工作流/Skill
```

**价值**：线上真实问题反哺离线调优，测试集越来越贴近实际。

---

### 结合点 3：总 Agent 统一入口

总 Agent 同时挂两个 MCP：

| MCP       | 工具                        | 何时用                       |
| --------- | --------------------------- | ---------------------------- |
| check-mcp | `check.evaluate`            | 每次用户提问（单次循环内）   |
| tune-mcp  | `tune.start` / `get_result` | 用户说「调到 90%」或定期任务 |

```text
普通提问     → 走单次循环（Skill 自动触发 check）
调优指令     → 走 Tune Engine（tune.start）
```

## 一次完整生命周期示例

```text
【第1周】只上单次循环
  用户问合同解析 → 第一次 70 分 → 重试 → 90 分输出
  同时记录失败 case

【第2周】部署 Tune Engine
  把第1周积累的 20 条失败 case 导入 benchmark
  tune.start → 工作流 v1→v3，通过率 72%→91%

【第3周】两者一起跑
  生产 Agent 用 v3 配置
  用户提问：第一次就 85 分，重试一次到 98 分（比以前少重试）
  新的失败 case 继续导入 Tune Engine

【第4周】定期调优
  每周自动 tune.start 一次
  有提升就晋升新版本
```

## 推荐配合方式

```text
日常（自动）  单次循环一直开着
              → 保证单次体验

定期（手动/定时） Tune Engine 跑一轮
              → 提升基线配置

持续（积累）  单次失败 case → benchmark
              → 测试集越来越真实
```

---

## 需要注意的一点

两者**不要抢同一件事**：

| 正确                                    | 错误                             |
| --------------------------------------- | -------------------------------- |
| 单次循环：当次重试，不改版本            | 单次循环里再去改工作流配置       |
| Tune Engine：改版本，不处理用户当次请求 | 用户每个 query 都触发 tune.start |

分工清楚就不会冲突。

## 二、接入要做的事（就 3 步）

### 步骤 1：确认平台「跑 Agent」接口

你们平台需要能提供类似能力：

```http
POST /api/agent/run    （路径以你们实际为准）

请求：
{
  "agent_id": "合同解析Agent",
  "query": "解析以下合同……"
}

响应：
{
  "answer": "总结模型最终输出",
  "plan_trace": [...]    // 可选
}
```

### 步骤 2：改 Tune Engine 对接代码

**文件 1：`tune-engine/app/config.py`**

```python
USE_MOCK_AGENT = False
AGENT_PLATFORM_URL = "http://你们平台地址:端口"
```

**文件 2：`tune-engine/app/clients/agent_platform.py`**

把 `RealAgentPlatformClient.run()` 改成你们真实 URL 和字段名，例如：

```python
def run(self, entry_ref: str, input_data: dict, ground_truth=None) -> dict:
    agent_id = entry_ref.split("@")[0]   # contract-parse-agent@v3 → contract-parse-agent
    query = input_data.get("query", "")

    resp = httpx.post(
        f"{self.base_url}/api/agent/run",   # 你们真实路径
        json={"agent_id": agent_id, "query": query},
        timeout=120,
    )
    data = resp.json()
    answer = data.get("answer", "")

    return {
        "status": "success",
        "output": self._parse_json_from_answer(answer),  # 从 answer 抽 JSON
        "summary": answer,
        "plan_trace": data.get("plan_trace", []),
        "latency_ms": data.get("latency_ms", 0),
        "cost_tokens": data.get("cost_tokens", 0),
    }
```

### 步骤 3：注册 Target + 测试集，启动服务

```bash
cd tune-engine
python scripts/init_demo.py          # 或注册你自己的 target
python -m uvicorn app.main:app --port 8100
```

## 三、手动 vs 自动：都能，而且建议组合用

Tune Engine **本身不会自动跑**，必须有人/有系统调它的 API。触发方式可以任选或组合：

| 方式                 | 类型   | 怎么做                                    | 适合场景             |
| -------------------- | ------ | ----------------------------------------- | -------------------- |
| **命令行**           | 手动   | `python scripts/run_demo.py`              | 开发验证             |
| **curl / API**       | 手动   | `POST /api/v1/tune/sessions`              | 运维、测试同学       |
| **Swagger UI**       | 手动   | 打开 `http://host:8100/docs` 点按钮       | 调试                 |
| **总 Agent + MCP**   | 半自动 | 用户说「把合同解析调到90%」→ `tune.start` | 业务人员自然语言触发 |
| **定时任务 cron**    | 自动   | 每周一 2:00 调 `tune.start`               | 定期优化             |
| **失败 case 达阈值** | 自动   | 积累 20 条失败 → 脚本触发                 | 线上问题驱动         |

## 五、`async` 选手动还是自动

| `async` | 行为                        | 建议                   |
| ------- | --------------------------- | ---------------------- |
| `false` | 调用方阻塞等到结束          | 手动调试、MCP 同步调用 |
| `true`  | 立刻返回 session_id，后台跑 | 定时任务、长时间调优   |

调优一轮可能要几分钟到几十分钟，**生产环境建议 `async: true`**，再用 `status` / `result` 轮询。

## 六、调优过程里谁控制什么

```text

  - 何时触发（手动 / 定时 / 对话）

  - 达标条件（success_criteria_override）
  - 最多几轮（constraints_override.max_tune_iters）

  - 批量调平台跑测试 query
  - 打分
  - 未达标 → 归因 → 打补丁 → 新版本
  - 达标 → 返回 optimal
  - 到上限 → 返回 best_effort

  - optimal 后是否晋升到生产（改平台里的 Agent/工作流版本）

```

## 三、平台每次跑的结果，能怎么用

假设平台每次返回类似：

```json
{
  "query": "解析以下合同……",
  "workflow_output": { "party_a": "甲公司", "amount": null },
  "answer": "甲方为甲公司，乙方为乙公司，金额未识别……",
  "plan_trace": [
    { "step": 1, "intent": "路由到合同解析工作流" },
    { "step": 2, "intent": "抽取字段" }
  ],
  "summary": "同上 answer"
}
```

可以拆成三种用途：

### 用途 A：当场判断要不要重试（方案一）

```text
工作流 output / answer
    → check.evaluate
    → passed=false → 记录 failure_points → 重规划再跑
```

用的是**当次**结果，不存库也能工作。

### 用途 B：存成样本，以后给 Tune Engine 用（方案二）

```text
这次 query + answer/output +（人工或规则补）ground_truth
    → 写入 failure_cases.jsonl
    → import_failure_cases.py
    → Tune Engine benchmark
    → tune.start 批量调优
```

用的是**历史**结果，改善以后所有请求。

### 用途 C：分析「总结」本身好不好（两层评估）

```text
第一层：字段对不对（evaluator 对 output/JSON）
第二层：总结是否完整、有无漏报（可用规则或 LLM 评 summary）
```

例如工作流抽对了，但总结漏了金额 → 该改**总结 prompt**，不是工作流。

## 四、能不能只靠「总结」来调优？

**能，但建议「总结 + 结构化结果」一起用。**

| 只有总结（自然语言） | 总结 + 工作流 output  |
| -------------------- | --------------------- |
| 可以评「说得全不全」 | 可以评「字段对不对」  |
| 难自动比对标准答案   | 容易做 pass_rate / F1 |
| 适合调总结 prompt    | 适合调工作流 + 总结   |

**推荐做法：**

```text
调优打分 = 0.7 × 字段准确率(output) + 0.3 × 总结质量(summary)
```

总结质量可以简单规则：是否包含所有字段名、是否声明缺失项等。

## 五、具体怎么实现（不改平台底层）

你们不能改平台代码，可以用 **MCP + 外挂收集服务**：

### 架构

```text
平台每次跑完
    ↓
总 Agent / 工作流 调 MCP：run.log（记录本次结果）
    ↓
收集服务（简单 Python，或 Tune Engine 扩展）
    ↓
├─ passed=false → failure_cases.jsonl
└─ 定期 → Tune Engine benchmark → tune.start
```

### MCP 工具示例：`run.log`

```json
// 请求
{
  "target_id": "contract-parse",
  "query": "原始问题",
  "output": { "party_a": "...", "amount": null },
  "summary": "总结模型全文",
  "plan_trace": [...],
  "expected_fields": ["party_a", "party_b", "amount", "sign_date"]
}

// 响应
{
  "passed": false,
  "score": 0.75,
  "failure_points": { "missing_fields": ["amount"] },
  "saved": true,
  "case_id": "online_20250624_001"
}
```

在 **auto-retry-replan Skill** 里加一条：Step2 检查完后，顺便调 `run.log` 把结果存下来。

- **方案一**：`check.evaluate` 决定是否重试  
- **数据回流**：`run.log` 存每次结果  
- **方案二**：定期从 jsonl 导入 Tune Engine

先说结论：**项目里没有叫 “Loop Engineering” 的模块**；设计里对应的是 **Optimizer（优化层）**，MVP 里被拆进 `orchestrator.py` + `patch_proposer.py`，没有单独做成一层服务。

## 1. 两个概念分别是什么

在对话里的架构图里，其实分的是这两块（当时没用 “Loop Engineering” 这个名字）：

| 概念                               | 对话里的叫法         | 干什么                                                      |
| ---------------------------------- | -------------------- | ----------------------------------------------------------- |
| **Loop Orchestrator**              | 控制层 Control Plane | **管循环**：何时跑、跑几轮、停不停、状态机、调度各模块      |
| **Loop Engineering**（若对应设计） | **优化层 Optimizer** | **管怎么变好**：归因 → 提案 → 打补丁 → 沙箱对比 → 晋升/回滚 |

闭环：

```text
Observe → Evaluate → Diagnose → Propose → Apply → Verify → (继续/结束)
         ↑___________________ Orchestrator 管整条链的节奏 ________↑
                              Engineering 管 Diagnose/Propose/Verify 怎么做得对
```

可以记成：

- **Orchestrator** = 指挥（指挥家）
- **Engineering（优化层）** = 改谱、试奏、判断要不要换版本（乐手 + 编曲）

---

## 2. 关系：谁调用谁

```text
Loop Orchestrator（orchestrator.py）
    │
    ├─ 调 Agent 跑 benchmark          ← Runtime
    ├─ 调 Evaluator 打分              ← Evaluation
    ├─ 调 diagnose / propose_patch    ← Engineering（优化）
    ├─ 决定 continue / stop           ← Policy
    └─ 写 session / iter 状态         ← 状态机
```

**Orchestrator 不决定「改什么」**，只决定「这一轮走完没有、要不要下一轮」。  
**Engineering 决定「差在哪、补丁是什么、新版本是否更好」**。

| 设计模块          | 规划位置 | 项目里有没有 | 实际文件                                   |

| Loop Orchestrator | 控制层   | ✅ 有         | `tune-engine/app/engine/orchestrator.py`   |
| Diagnoser         | 优化层   | ⚠️ 合并       | `patch_proposer.py` 里的 `diagnose()`      |
| Patch Proposer    | 优化层   | ⚠️ 合并       | `patch_proposer.py` 里的 `propose_patch()` |
| Shadow Compare    | 优化层   | ❌ 未实现     | 仅在 spec 里；MVP 直接晋升                 |
| Version Registry  | 控制层   | ⚠️ 简化       | `config_version` 表 + Mock `apply_patch`   |

spec 里规划的目录（`05-api-and-directory.md`）：

```text
engine/
  orchestrator.py      ← 已实现
  diagnoser.py         ← 未单独建文件
  patch_proposer.py    ← 已实现（含 diagnose）
  shadow_compare.py    ← 未实现
```

所以：**不是没用 “Engineering” 的能力，而是没单独起名、没拆成独立服务。**

## 4. 为什么 MVP 没用独立的 “Loop Engineering”

### 原因 1：术语不同

设计图写的是 **Optimizer / 优化层**，没有 **Loop Engineering** 这个模块名。实现时自然只做了 `orchestrator` + `patch_proposer`。

### 原因 2：MVP 优先「能跑通」

目标是最快看到：

```text
跑测 → 打分 → 改配置 → 再跑 → optimal
```

把 orchestrator 和 engineering 写进一个主循环文件，比拆 4～5 个服务更快。

### 原因 3：Engineering 依赖平台能力，当时没有

完整 Engineering 需要：

- 真实 `apply_patch`（改 workflow/skill 版本）
- `shadow_compare`（baseline vs candidate）
- 可选 LLM 归因

你们当时还不能改平台底层，所以 engineering 只能做**规则级** `diagnose` + **模拟** `apply_patch`，拆独立层收益不大。

### 原因 4：Shadow 被刻意跳过

`orchestrator.py` 里写明：

```python
# MVP：补丁后直接晋升（真实环境应 shadow_compare）
iter_rec.shadow_compare_json = {"improved": True, "note": "MVP 跳过 shadow，直接晋升"}
```

没有可靠的 shadow，**独立 Engineering 层就不完整**，先并入 orchestrator 更简单。

### 原因 5：在线单次循环不在 Tune Engine 里

「单次 query 重试」放在 **Skill + check-mcp**，不算 Tune Engine 的 Loop Engineering，进一步减少了要拆的模块数。

---

## 5. 和单次循环的分工（避免混淆）

| 循环类型                | Orchestrator 角色             | Engineering 角色                                         |
| ----------------------- | ----------------------------- | -------------------------------------------------------- |
| **Tune Engine（离线）** | `orchestrator.py` 管多轮 iter | `patch_proposer.py` 管补丁                               |
| **单次循环（在线）**    | 平台 5 步规划（Skill 约束）   | `check.evaluate` + 重规划 prompt，**没有** tune 的 patch |

在线侧没有「改配置版本」的 Engineering，只有「当次重试」。

---

## 6. 若以后要补全 Engineering 层

可以按 spec 拆成：

```text
orchestrator.py        只保留：状态机、调度、stop policy
diagnoser.py           失败归因（可接 LLM）
patch_proposer.py      已有
shadow_compare.py        baseline vs candidate
version_registry.py    晋升 / 回滚
```

Orchestrator 变成「薄编排」，Engineering 变成可插拔插件——**这是 spec 里的目标形态，不是当前 MVP 形态。**

```64:74:check-mcp/server.py
def run_evaluate(arguments: dict[str, Any]) -> dict[str, Any]:
    task_type = arguments.get("task_type", "contract-parse")
    output = parse_output(arguments.get("output", {}))
    expected_fields = arguments.get("expected_fields", [])
```

## check-mcp 外挂的目的是什么？

核心就一件事：**给工作流输出做确定性、可复现的自动质检**。

| 目的               | 说明                                                         |
| ------------------ | ------------------------------------------------------------ |
| **客观判分**       | 字段齐不齐、`sign_date` 格式对不对、`amount` 是不是数字——用规则算，不靠模型「感觉」 |
| **结构化失败信息** | 返回 `failure_points`（缺哪些字段、哪项格式错），供下一轮重规划直接用 |
| **统一评分标准**   | 与 `run-log-mcp`、`Tune Engine` 共用 `shared/evaluators.py`，线上检查和离线调优口径一致 |
| **不改平台底层**   | 平台只挂 MCP，不用改规划引擎或工作流执行器                   |

在单次循环里的位置：

```
Skill 教规划：第 2 步要检查、没过要重试
        │
        ▼
工作流执行 → 产出 output
        │
        ▼
check-mcp.check.evaluate  ← 外挂：硬规则打分
        │
        ├─ passed=true  → Skill 说可以总结了
        └─ passed=false → failure_points 带回 Step 3 重跑
```

没有 check-mcp，规划模型只能自己「看一眼觉得行不行」——**不稳定、不可追溯、也没法和 Tune Engine 对齐**。

## 能放到 Skill 里吗？

**循环逻辑可以，评估执行不行。**

### Skill 已经承担的部分（可以、也应该在 Skill 里）

`auto-retry-replan` Skill 负责的是**行为策略**：

- 5 步怎么分配（执行 → 检查 → 重试 → 再检查 → 总结）
- 什么时候调 `check.evaluate`
- `passed=false` 时怎么把 `failure_points` 写进重试 query
- 什么时候停止、怎么输出 `optimal` / `best_effort`

这些**本来就是 Skill 的工作**，不需要再塞进 MCP。

### 不适合放进 Skill 的部分（必须靠 MCP 或类似服务）

| 能力                     | 放 Skill？ | 原因                                                     |
| ------------------------ | ---------- | -------------------------------------------------------- |
| 检查字段是否缺失         | ❌          | Skill 是 Prompt，不能跑 Python、不能 garant 每次结果一致 |
| 校验日期/金额格式        | ❌          | 需要确定性规则，不是让模型「猜」                         |
| 输出固定 JSON 结构       | ❌          | 模型输出格式不稳定                                       |
| 和 Tune Engine 同一套分  | ❌          | 离线调优要可复现的 benchmark 评分                        |
| 写 `failure_cases.jsonl` | ❌          | 需要 `run-log-mcp` 落盘，Skill 做不了 I/O                |

如果把评估写进 Skill，等价于让规划模型自己判分：

```text
「请检查 output 是否包含 party_a、party_b、amount、sign_date…」
```

常见问题：

- 同一份 output，不同次可能判得不一样  
- 没有稳定的 `score`、`field_score`  
- `failure_points` 结构不固定，重试质量差  
- 无法和离线 Tune 共用评分逻辑

```
Skill        → 教 Agent「怎么循环」（软策略，Prompt）
check-mcp    → 告诉 Agent「过没过」（硬规则，工具）
```

- **Skill** = 操作手册（「做完要质检，不合格返工」）

- **run-log-mcp** = 质检台账（记下来以后分析）

| **A. 纯 Skill**       | 让规划/总结模型自己检查                              | 不稳定，无法对接 Tune，不推荐      |
| **B. 平台工作流节点** | 在工作流里加「检查」节点（若平台支持自定义代码/API） | 每个业务工作流都要改，不能通用复用 |
| **C. 合并 MCP**       | 把 check + run.log 合成一个 MCP                      | 仍要挂 MCP，只是少一个地址         |

当前架构选 **Skill + 两个 MCP**，是因为你们**不能改平台底层**，又要**通用接入、评分一致**。

---

**结论：** check-mcp 的目的是提供**可复现的自动质检**；循环策略放 Skill，评估执行放 MCP。Skill 能写「何时检查、如何用结果重试」，但不能替代 `check.evaluate` 的规则引擎和结构化输出。

## 1. 为什么能通用？

你们平台本身已经提供了可组合的积木：

```
Agent 引用 Agent
工作流 引用 工作流
挂 Skill（教规划怎么做事）
挂 MCP（可调外部工具）
```

通用接入**不是改平台**，而是在平台积木之上，固定三层分工：

| 层           | 谁维护       | 干什么                                                     | 是否随业务变 |
| ------------ | ------------ | ---------------------------------------------------------- | ------------ |
| **基础设施** | 运维部署一次 | check-mcp / run-log-mcp / tune-engine                      | ❌ 不变       |
| **通用壳**   | 项目提供模板 | retry-assistant、wf_retry_wrapper、auto-retry-replan Skill | ❌ 不变       |
| **业务配置** | 各团队自己填 | task_type、expected_fields、workflow_ref                   | ✅ 每业务不同 |

新业务（合同、发票、报表）**只改第三层配置**，前两层原样复用。

## 2. 通用接入原理（分层解耦）

```mermaid
flowchart TB
    subgraph Platform["你们的 Agent 平台（不改底层）"]
        MA[总 Agent]
        BA[业务 Agent / 工作流]
        SK[Skill 循环策略]
        MA -->|引用| RA[在线重试助手]
        MA --> BA
        BA --> SK
    end

    subgraph Config["配置层（每业务不同）"]
        TR[task-registry.yaml<br/>task_type + expected_fields]
    end

    subgraph Services["外挂服务（部署一次）"]
        CK[check-mcp<br/>规则评分]
        RL[run-log-mcp<br/>落盘记录]
        TE[tune-engine<br/>批量调优]
    end

    SK -->|调 MCP| CK
    SK -->|调 MCP| RL
    TR -.->|告诉 Skill 评什么字段| SK
    RL -->|failure_cases| TE
```

**关键点：**

1. **Skill 管流程**（何时执行、检查、重试、总结）——通用，所有业务共用一份 `auto-retry-replan`
2. **MCP 管能力**（打分、写日志）——通用，接口固定 `check.evaluate` / `run.log`
3. **task-registry 管差异**（评哪些字段、路由哪个工作流）——每业务只加一段 YAML

## 3. 「不用改 MCP 代码」怎么做到？

check-mcp 内部用 **`task_type` 分发**：

```python
EVALUATORS = {
    "contract-parse": evaluate_contract,
    "invoice-parse": evaluate_invoice,  # 将来加
}
```

调用时传：

```json
{
  "task_type": "contract-parse",
  "output": { ... },
  "expected_fields": ["party_a", "party_b", "amount", "sign_date"]
}
```

**业务差异全部通过参数表达**，不靠每个团队 fork 一份 MCP。

`task-registry.yaml` 就是这份参数的登记表：

```yaml
invoice-parse:
  task_type: invoice-parse
  expected_fields: [seller, buyer, amount, tax_id, invoice_date]
  workflow_ref: wf_invoice_parse
  trigger_keywords: [发票, 开票]
```

规划模型看到「发票」→ 用 invoice 的 `expected_fields` 调同一个 `check.evaluate`。

## 4. 别人怎么接？（3 选 1，原理相同）

| 方式                    | 做法                            | 本质                                       |
| ----------------------- | ------------------------------- | ------------------------------------------ |
| **A. 引用 Agent**       | 总 Agent 引用 `retry-assistant` | 把 Skill + 两个 MCP 打包成一个可引用 Agent |
| **B. 引用工作流**       | 业务先走 `wf_retry_wrapper`     | 把循环逻辑做成可引用工作流壳               |
| **C. 直接挂业务 Agent** | 现有 Agent 加 Skill + 两个 MCP  | 不新建 Agent，能力直接贴到业务上           |

三种方式**底层链路一样**：

```
执行工作流 → check.evaluate → run.log → 不过就带 failure_points 重跑
```

区别只是：**通用壳放在 Agent 里、工作流里，还是业务 Agent 自己身上。**

## 5. 和「写死在合同 Agent 里」的区别

|          | 写死方案                       | 通用接入                               |
| -------- | ------------------------------ | -------------------------------------- |
| 重试逻辑 | 每个业务 Agent 复制一份 Prompt | 共用 `auto-retry-replan` Skill         |
| 评分     | 每个业务自己写规则 / 让模型猜  | 共用 check-mcp，按 task_type 分发      |
| 新业务   | 改 Agent + 可能改代码          | 加 task-registry 一行 + 自己的工作流   |
| 离线调优 | 各业务评分口径不一致           | run-log → Tune Engine 同一套 evaluator |

## 6. 接入方实际要做什么？

**运维（一次）：**

```bash
bash scripts/start-all.sh   # 或 docker compose up -d
```

**业务方（每个新业务）：**

1. `YOUR_HOST` 改成实际 IP
2. 平台引用 `retry-assistant`（或挂 Skill + MCP）
3. `task-registry.yaml` 加一段 `expected_fields` + `workflow_ref`
4. （可选）Tune Engine 注册 target + benchmark

**不需要：** 写 Python、改 check-mcp 源码、改平台底层。

## 7. 一句话总结

> **通用接入 = 平台「引用 + Skill + MCP」组合能力，外挂服务提供固定工具接口，业务差异全部收敛到 `task_type` / `expected_fields` 配置。**

Skill 解决「怎么循环」，MCP 解决「怎么客观打分和落盘」，task-registry 解决「评什么、走哪个工作流」——三者解耦，所以才能给任意业务复用。

这两个是**合同解析**场景里，我们约定要从合同里抽出来的 **JSON 字段名**（不是技术术语）。

## 在本项目里指什么

| 字段          | 英文含义 | 合同里对应什么 | 示例                                |
| ------------- | -------- | -------------- | ----------------------------------- |
| **amount**    | 金额     | 合同金额、价款 | `100000` 或 `10万元`                |
| **sign_date** | 签订日期 | 合同签署日期   | `"2026-01-15"`（要求 `YYYY-MM-DD`） |

连同另外两个字段，合同解析一共要抽 4 个：

```yaml
expected_fields:
  - party_a      # 甲方
  - party_b      # 乙方
  - amount       # 金额
  - sign_date    # 签订日期
```

定义在 `integrations/task-registry.yaml` 的 `contract-parse` 里。

---

## 和检查 API 的关系

`check-mcp` 会检查：

1. **amount** 有没有、能不能转成数字  
2. **sign_date** 有没有、是不是 `YYYY-MM-DD` 格式（如 `2026-01-15`）

所以演示里「第一次缺 amount、sign_date」→ `passed=false`；补全后 → `passed=true`。
