# 批量 Tune Engine 调优 vs 真实案例调优 — 对比与示例

> 配合 `run-log-mcp` 使用：线上真实案例自动收集，批量调优消费这些案例。

---

## 一、两种调优方式一句话

| 方式 | 一句话 |
|------|--------|
| **批量 Tune Engine** | 用**事先准备好的一批测试题**，离线自动改配置，把通过率调到目标值 |
| **真实案例调优** | 用**平台每次真实跑出来的 query/output/summary**，积累失败样本，再交给 Tune Engine 调 |

二者**不矛盾**：真实案例最终也是进 Tune Engine，只是数据来源不同。

---

## 二、对比总表

| 维度 | 批量 Tune Engine | 真实案例（run-log-mcp） |
|------|------------------|-------------------------|
| **数据来源** | 人工编写的 benchmark | 平台每次运行的 output + summary |
| **何时产生** | 上线前 / 专项测试时准备 | 用户每次提问后自动产生 |
| **标准答案** | 事先写好 `ground_truth` | 多数需事后补；有则可直接导入 |
| **触发调优** | 手动 / 定时 / MCP `tune.start` | 先收集，再定期 `tune.start` |
| **评分方式** | evaluator 逐字段比对 | run.log 规则评分（字段+总结） |
| **覆盖场景** | 已知难点、标准合同样本 | 线上真实分布、长尾 case |
| **优点** | 标准统一、可复现、可回归 | 贴近生产、持续更新、发现真实问题 |
| **缺点** | 可能与真实使用脱节 | 需补 ground_truth、有噪音 |
| **典型用途** | 冷启动、发版前回归 | 上线后持续进化 |

---

## 三、三者关系（含单次循环）

```text
                    用户 query
                        │
                        ▼
              ┌─────────────────────┐
              │ 方案一：单次循环      │
              │ check → 重试 → 总结  │
              └─────────┬───────────┘
                        │ 每次跑完

              ┌─────────────────────┐

          ▼                           ▼
   run_logs.jsonl              failure_cases.jsonl

          │                           │
          │                           ▼

          │                           │
          └─────────────┬─────────────┘
                        ▼
              ┌─────────────────────┐
              │ Tune Engine 批量调优  │
              │ + 人工 benchmark 合并 │
              └─────────────────────┘
```

## 四、案例 A：批量 Tune Engine（人工测试集）

### 场景

上线前，用 15 份标准合同样本，把合同解析调到通过率 ≥ 90%。

### 测试数据（人工编写）

```json
{
  "case_id": "bench_01",
  "input": {
    "query": "解析以下合同，输出JSON：party_a, party_b, amount, sign_date\n甲方：北京甲科技有限公司\n乙方：上海乙贸易有限公司\n金额：50万元\n签署：2024-01-15"
  },
  "ground_truth": {
    "party_a": "北京甲科技有限公司",
    "party_b": "上海乙贸易有限公司",
    "amount": 500000,
    "sign_date": "2024-01-15"
  }
}
```

### 操作步骤

```bash
# 1. 注册 target
curl -X POST http://127.0.0.1:8100/api/v1/targets \
  -H "Content-Type: application/json" \
  -d @tune-engine/examples/contract-parse-target.json

# 2. 导入人工 benchmark
curl -X POST http://127.0.0.1:8100/api/v1/benchmarks/contract_bench_v1/cases \
  -H "Content-Type: application/json" \
  -d @你的cases.json

# 3. 触发批量调优
curl -X POST http://127.0.0.1:8100/api/v1/tune/sessions \
  -H "Content-Type: application/json" \
  -d '{"target_id": "contract-parse", "async": false}'
```

### 运行过程

```text
iter1: v1, pass_rate=35%  → 补丁 skill   → v2
iter2: v2, pass_rate=68%  → 补丁 workflow → v3
iter3: v3, pass_rate=92%  → optimal ✓
```

### 特点

- 15 条样本每条都有标准答案
- 同样本反复跑，对比版本提升
- 适合：**冷启动、发版回归、验收**

## 五、案例 B：真实案例（run-log-mcp 驱动）

### 场景

上线后，用户每天问合同解析。系统自动记录每次 output/summary，失败 case 积累后每周调优一次。

### 一次真实运行

**用户 query：**

```text
解析这份合同，输出 party_a, party_b, amount, sign_date：
甲方：广州丙建设集团  乙方：深圳丁装饰公司  金额：280万  签署：2023年11月8日
```

**平台跑出（Step 1）：**

```json
{
  "output": {
    "party_a": "广州丙建设集团",
    "party_b": "深圳丁装饰公司",
    "amount": null,
    "sign_date": "2023-11-08"
  },
  "summary": "已识别甲乙双方及签署日期，未能识别合同金额"
}
```

**Step 2：check.evaluate + run.log**

```bash
# check.evaluate → passed=false, missing amount
# run.log → 写入 run_logs.jsonl + failure_cases.jsonl
```

**run.log 返回：**

```json
{
  "logged": true,
  "case_id": "run_20250624_a1b2c3d4",
  "passed": false,
  "score": 0.68,
  "saved_to": ["run_logs.jsonl", "failure_cases.jsonl"],
  "tune_import_ready": false,
  "note": "无 ground_truth 的失败记录需人工补标准答案后再导入 Tune Engine"
}
```

**Step 3～4：重试成功，第二次 run.log passed=true，用户拿到完整结果。**

### 运营补标准答案（failure_cases.jsonl 中该条）

```json
{
  "case_id": "run_20250624_a1b2c3d4",
  "input": {"query": "解析这份合同……"},
  "ground_truth": {
    "party_a": "广州丙建设集团有限公司",
    "party_b": "深圳丁装饰工程有限公司",
    "amount": 2800000,
    "sign_date": "2023-11-08"
  },
  "tags": ["online", "failed", "contract-parse"],
  "tune_import_ready": true
}
```

### 每周批量调优

```bash
# 导入本周真实失败 case（已补 ground_truth 的）
cd tune-engine
python scripts/import_failure_cases.py \
  --file ../data/failure_cases.jsonl \
  --suite-id contract_bench_v1

# 合并人工 benchmark 后一起调优
curl -X POST http://127.0.0.1:8100/api/v1/tune/sessions \
  -d '{"target_id": "contract-parse", "async": true}'
```

### 特点

- 数据来自真实用户 query
- 先服务用户（单次重试），再沉淀数据
- 适合：**上线后持续优化、发现长尾问题**

## 六、案例 C：联合使用（推荐生产方案）

### 组合策略

```text
benchmark = 人工核心集（15条） + 真实失败集（每周新增，已补答案）
```

| 数据集 | 条数 | 权重 | 作用 |
|--------|------|------|------|
| 人工核心集 | 15 | 1.0 | 回归基准，防退化 |
| 真实失败集 | 5～20/周 | 1.2～1.5 | 针对线上真实问题 |

### 一周运行节奏

| 时间 | 动作 | 方式 |
|------|------|------|
| 实时 | 用户提问 → 单次循环 → run.log | 真实案例 |
| 每天 | 看 `run.stats` 通过率 | `curl run-log-mcp` |
| 周三 | 补 3～5 条 failure 的 ground_truth | 人工 |
| 周五 | import + tune.start | 批量 Tune Engine |
| 周五晚 | 评审 optimal 是否晋升 vN | 人工 |

### 示例：周五调优结果

```text
benchmark 合计 22 条（15 人工 + 7 真实失败已补答案）

tune.start 结果：
  iter1: v3, pass=76%
  iter2: v4, pass=88%
  iter3: v5, pass=93%  → optimal

晋升 v5 后：
  - 线上 amount 漏抽明显减少
  - 单次循环重试次数从平均 1.8 次 → 0.6 次
```

```bash
# 调优（手动或 cron）
curl -X POST .../api/v1/tune/sessions -d '{"target_id":"contract-parse"}'
```

## 八、何时用哪种

```text
只有人工测试集、还没上线        → 批量 Tune Engine（案例 A）
已上线、有真实流量               → run-log-mcp + 定期 Tune（案例 B）
生产环境长期运行               → 联合方案（案例 C）★ 推荐
用户单次提问要当场答好          → 方案一单次循环（与调优无关但配合 run.log）
```

## 九、文件与数据流索引

| 文件/服务 | 作用 |
|-----------|------|
| `run-log-mcp/server.py` | 收集 output/summary，评分，写 jsonl |
| `data/run_logs.jsonl` | 全部真实运行记录 |
| `data/failure_cases.jsonl` | 失败记录，待补答案后导入 |
| `tune-engine/scripts/import_failure_cases.py` | 导入 Tune Engine |
| `tune-engine/scripts/run_demo.py` | 批量调优演示（人工集） |
| `skills/auto-retry-replan/SKILL.md` | 含 run.log 必调规范 |

## 十、快速命令

```bash
# 启动收集服务
cd run-log-mcp && python server.py

# 查看近期通过率
curl -X POST http://127.0.0.1:8300/mcp/tools/call \
  -H "Content-Type: application/json" \
  -d '{"tool":"run.stats","arguments":{"target_id":"contract-parse"}}'

# 手动记一条（测试）
curl -X POST http://127.0.0.1:8300/mcp/tools/call \
  -H "Content-Type: application/json" \
  -d '{
    "tool": "run.log",
    "arguments": {
      "target_id": "contract-parse",
      "query": "解析合同……",
      "output": {"party_a":"甲公司","party_b":"乙公司"},
      "summary": "缺少金额和日期",
      "expected_fields": ["party_a","party_b","amount","sign_date"]
    }
  }'

# 导入并调优
cd tune-engine
python scripts/import_failure_cases.py --file ../data/failure_cases.jsonl
python scripts/run_demo.py
```
