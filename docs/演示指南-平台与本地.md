> **适用：** 平台 MCP/API 未通时用本地演示；平台联调通后按平台版走同一套用例。  
> **标杆场景：** 合同解析（`contract-parse`）  
> **仓库路径：** `/Users/wanghanbing/Documents/文本资料/l`

---

## 一、两种演示方式对照

| 维度 | 本地演示 | 平台演示 |
|------|----------|----------|
| **前提** | 不用 Agent 平台 | 平台工作流/API 节点可用 |
| **启动什么** | check-mcp + run-log-mcp（+ 可选 single-loop-local / tune-engine） | 同上服务 + 平台配工作流 |
| **谁执行合同解析** | curl 模拟 output，或 single-loop-local 调模型 | 平台业务工作流（合同解析 Agent） |
| **谁做检查/记录** | 手动 curl API，或 single-loop-local 自动 | 工作流 API 节点自动调 |
| **谁做重试** | 手动第二轮 curl，或 single-loop-local 自动 | Skill 教规划 / 工作流条件分支 |
| **能证明什么** | 检查逻辑、落盘、循环概念 | 端到端用户提问自动重试 |
| **当前状态** | ✅ 现在就能演示 | ⚠️ 待联调 |

**核心：** 本地验证的是**同一套 API 和业务规则**；平台通了只是把 curl 换成工作流节点。

## 二、典型案例（两个演示共用）

### 案例 A：第一次缺字段（应 `passed=false`）

**用户 query：**
```text
解析以下合同，输出 JSON：party_a, party_b, amount, sign_date

甲方：北京甲科技有限公司
乙方：上海乙贸易有限公司
（合同正文未写金额和日期）
```

**模拟工作流 output（第一次）：**
```json
{
  "party_a": "北京甲科技有限公司",
  "party_b": "上海乙贸易有限公司"
}
```

**期望 check 结果：**
- `passed: false`
- `failure_points.missing_fields`: `["amount", "sign_date"]`
- `recommendation`: `补全缺失字段后重试`

---

### 案例 B：重试后补全（应 `passed=true`）

**重试 query（带 failure_points）：**
```text
【原始任务】解析合同…

【上次执行问题】
{"missing_fields": ["amount", "sign_date"], ...}

【重试要求】必须补全 amount、sign_date
```

**模拟工作流 output（第二次）：**
```json
{
  "party_a": "北京甲科技有限公司",
  "party_b": "上海乙贸易有限公司",
  "amount": 100000,
  "sign_date": "2026-01-15"
}
```

**期望 check 结果：**
- `passed: true`
- `score` ≥ 0.8

---

### 案例 C：格式错误（可选演示）

```json
{
  "party_a": "甲公司",
  "party_b": "乙公司",
  "amount": "十万元",
  "sign_date": "2026/01/15"
}
```

**期望：** `passed: false`，`failures` 含 `format_error`（金额非数字、日期格式错）。

### 3.3 本地演示流程 A：手动 API（推荐汇报用）

**对应平台里的：** 工作流执行 → API 检查 → API 记录 → 条件重试

#### Step 0：说明

> 下面用 curl 代替平台工作流。平台通后，这两步变成工作流里的 HTTP 节点，URL 和 body 不变。

#### Step 1：模拟第一次工作流输出 → 检查

```bash
curl -s -X POST http://127.0.0.1:8200/api/v1/check/evaluate \
  -H 'Content-Type: application/json' \
  -d '{
    "task_type": "contract-parse",
    "output": {
      "party_a": "北京甲科技有限公司",
      "party_b": "上海乙贸易有限公司"
    },
    "expected_fields": ["party_a", "party_b", "amount", "sign_date"],
    "summary": "已识别甲乙双方，缺少金额和日期"
  }' | python3 -m json.tool
```

**讲解点：** `passed=false`，`missing_fields` 就是重试依据。

#### Step 2：记录第一次运行

```bash
curl -s -X POST http://127.0.0.1:8300/api/v1/run/log \
  -H 'Content-Type: application/json' \
  -d '{
    "target_id": "contract-parse",
    "task_type": "contract-parse",
    "query": "解析合同：北京甲科技有限公司、上海乙贸易有限公司…",
    "output": {
      "party_a": "北京甲科技有限公司",
      "party_b": "上海乙贸易有限公司"
    },
    "summary": "已识别甲乙双方，缺少金额和日期",
    "expected_fields": ["party_a", "party_b", "amount", "sign_date"],
    "attempt": 1,
    "check_result": {
      "passed": false,
      "score": 0.5,
      "failure_points": {
        "missing_fields": ["amount", "sign_date"]
      }
    }
  }' | python3 -m json.tool
```

**讲解点：** `saved_to` 含 `failure_cases.jsonl`，供 Tune Engine 导入。

#### Step 3：模拟重试后 output → 再检查

```bash
curl -s -X POST http://127.0.0.1:8200/api/v1/check/evaluate \
  -H 'Content-Type: application/json' \
  -d '{
    "task_type": "contract-parse",
    "output": {
      "party_a": "北京甲科技有限公司",
      "party_b": "上海乙贸易有限公司",
      "amount": 100000,
      "sign_date": "2026-01-15"
    },
    "expected_fields": ["party_a", "party_b", "amount", "sign_date"],
    "summary": "party_a 北京甲科技 party_b 上海乙贸易 amount 100000 sign_date 2026-01-15"
  }' | python3 -m json.tool
```

**讲解点：** `passed=true` → 进入总结（平台里由 Skill 驱动 Step 5）。

#### Step 4：看统计（可选）

```bash
curl -s "http://127.0.0.1:8300/api/v1/run/stats?target_id=contract-parse" | python3 -m json.tool
tail -n 2 ../data/run_logs.jsonl
```

## 四、平台演示（联调通后 — 目标形态）

### 4.1 演示目标

用户在前端发一条 query，平台自动：执行 → 检查 → 记录 → 重试 → 总结。

### 4.2 启动什么

**服务端（与本地相同）：**
```bash
bash scripts/start-all.sh
# 或分别启动 check-mcp :8200、run-log-mcp :8300
```

**平台侧配置一次：**

| 配置项 | 值 |
|--------|-----|
| 引用工作流 | `wf_retry_wrapper_api`（见 `integrations/workflows/wf_retry_wrapper_api.json`） |
| Skill | `auto-retry-replan-api` |
| check API | `POST http://HOST:8200/api/v1/check/evaluate` |
| run.log API | `POST http://HOST:8300/api/v1/run/log` |
| 业务工作流 | `wf_contract_parse`（你们的合同解析） |
| task_type | `contract-parse` |
| expected_fields | `party_a, party_b, amount, sign_date` |

`YOUR_HOST` 改为服务器实际 IP。

---

### 4.3 平台演示流程

```text
┌──────────┐
│  用户     │  输入案例 A 的 query（故意缺金额日期信息的合同文本）
└────┬─────┘
     ▼
┌──────────────────────────────────────────┐
│ 总 Agent / 循环调优壳工作流                │
│  Step1 → 合同解析工作流 → output（缺字段） │
│  Step2 → API check → passed=false         │
│  Step3 → API run.log → 落盘               │
│  Step4 → 带 failure_points 重跑工作流     │
│  Step5 → API check → passed=true          │
│  Step6 → API run.log                      │
│  Step7 → 总结 optimal                     │
└──────────────────────────────────────────┘
     ▼
┌──────────┐
│  用户     │  看到完整 JSON + 说明尝试了 2 次
└──────────┘
```

#### Step 1：准备测试输入（案例 A）

在平台对话框输入：

```text
解析以下合同，输出 JSON 字段 party_a, party_b, amount, sign_date：

甲方：北京甲科技有限公司
乙方：上海乙贸易有限公司
本合同约定双方合作事宜。（故意不写金额和签订日期）
```

#### Step 2：观察第一次执行

- 工作流返回 output（可能只有 party_a、party_b）
- 工作流 API 节点调 check → 应 `passed=false`
- 工作流 API 节点调 run.log → `logged: true`

#### Step 3：观察自动重试

- Skill 驱动规划带 `failure_points` 重跑
- 第二次 output 应补全 amount、sign_date
- 第二次 check → `passed=true`

#### Step 4：验证落盘（给开发看）

```bash
curl "http://HOST:8300/api/v1/run/stats?target_id=contract-parse"
# 或服务器上
tail data/run_logs.jsonl
```

#### Step 5（可选）：换业务只改配置

把 `task_type` 改成 `invoice-parse`，换发票 query，证明通用性（不需改 API 服务）。

### 4.4 平台 vs 本地 — 同用例对照

| 环节 | 本地现在怎么做 | 平台通了怎么做 |

## 七、演示话术模板

**开场：**
> 今天演示的是 Agent 外层循环调优。平台底层没改，我们在外面加了检查和记录两个 Flask 服务。平台联调通前，本地用同样 API 验证逻辑。

**案例 A 后：**
> 第一次解析缺 amount 和 sign_date，检查 API 返回结构化 failure_points，不是模型自己说「我觉得不对」。

**案例 B 后：**
> 重试补全后 passed=true。平台里这一步由 Skill 和工作流自动完成。

**收尾：**
> 本地验证的是同一套 API。平台通了以后，工作流 API 节点调相同地址，别人引用我们的模板和 Skill 即可接入。

## 八、故障排查

| 现象 | 处理 |
|------|------|
| curl 连接拒绝 | 确认 `python app.py` 在 8200/8300 运行 |
| passed 总是 false 且 summary_score=0 | 检查请求里是否传了 `summary` |
| run.log 400 | 缺 `target_id` 或 `query` |
| single-loop-local 模型报错 | 检查 `config.yaml` 的 base_url / 鉴权 |
| 平台 API 节点 500 | 查 URL 是否含 `/api/v1/...`，Content-Type 是否 json |
