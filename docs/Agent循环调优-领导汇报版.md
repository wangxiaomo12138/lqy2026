# 二、第 1 页：背景（给领导）

## 痛点

| 问题 | 现状 | 影响 |
|------|------|------|
| 单次答不好 | 合同解析常缺字段就直接输出 | 用户体验差 |
| 长期难提升 | 改配置没有客观评测 | 不知道改得有没有用 |
| 失败无沉淀 | 坏 case 没入库 | 无法系统性优化 |

## 约束

- 只能**接入** Agent 平台（挂 Skill、工作流、API）
- **不能改**平台底层规划/执行引擎

## 目标

> 让用户每次提问**自动重试变好**，让工作流配置**批量迭代变优**。

# 三、第 2 页：两层循环（核心图）

```text
┌──────────────────────────────────────┐
│ 内层：单次循环（每次用户提问）         │
│ 执行 → 检查 → 不过重试 → 最优输出     │
│ 解决：这一次答得好                    │
└─────────────────┬────────────────────┘
                  │ 失败记录
                  ▼
┌──────────────────────────────────────┐
│ 外层：Tune Engine（离线批量）         │
│ 跑分 → 改配置 → 新版本 → 再跑         │
│ 解决：长期能力持续提升                │
└──────────────────────────────────────┘
```

**不是二选一，是互补。**

# 五、第 4 页：现行推荐方案（给开发）

## 标准包（别人引用即用）

```text
循环调优标准包
├── Skill: auto-retry-replan-api
├── 工作流: wf_retry_wrapper_api（API 每步已写好）
├── check-mcp  :8200  Flask  POST /api/v1/check/evaluate
└── run-log    :8300  Flask  POST /api/v1/run/log
```

## 单次提问怎么走

```text
用户提问
  → 业务工作流（合同解析等）
  → API 检查（字段齐不齐、格式对不对）
  → API 记录（成功失败都落盘）
  → 不过？带问题描述重跑
  → 总结输出
```

## 分工

| 组件 | 干什么 |
|------|--------|
| Skill | 教 Agent 何时检查、何时重试 |
| check API | 客观打分（规则引擎，不靠模型猜） |
| run-log API | 记录每次结果 |
| 业务工作流 | 真正执行任务 |

**新业务只改配置（评哪些字段），不改代码。**

# 八、可直接念的总结（30 秒）

> 我们在 Agent 平台外层建设循环调优：内层让用户每次提问自动检查重试，外层用 Tune Engine 批量提升配置。实现上不修改平台底层，通过 Skill 规定策略、Flask API 提供客观检查和数据落盘。  
> 方案设计、服务 MVP、接入模板均已完成；平台 MCP 联调遇 500，已改为 **API + Flask** 路径并保留本地验证方案。  
> 建议以合同解析为标杆，优先打通平台 API 节点联调，同步推进离线 Tune 接真实 API。

# 九、常见问题（Q&A 备答）

**Q：现在业务能直接用吗？**  
A：服务可部署、API 可测；需平台工作流 API 节点联调通过后，业务方引用模板即可。

**Q：为什么不用 MCP？**  
A：仍保留 MCP SSE 实现；平台当前 500，API 是更稳的替代，底层评分逻辑相同。

**Q：和现有合同解析 Agent 关系？**  
A：在外面包一层检查重试，不替换原 Agent，原工作流仍负责解析。

**Q：通用吗？**  
A：靠 task_type + expected_fields 配置，合同、发票等业务只改配置表。

**Q：投入多少人了？**  
A：方案 + MVP 一人可维护；联调需平台侧配合。
